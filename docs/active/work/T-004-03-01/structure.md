## `Dockerfile.session`

### Stages

Stage `node-runtime`:

```dockerfile
FROM node:24.18.0-bookworm-slim AS node-runtime
```

It supplies exact Node/npm files under `/usr/local` for the target architecture.

Final stage:

```dockerfile
FROM docker.io/cloudflare/sandbox:0.12.3
```

It owns the final filesystem and the unchanged `/container-server/sandbox` entrypoint.

### Build arguments

```text
TARGETARCH                 supplied by Docker/BuildKit
CODE_SERVER_VERSION       default 4.127.0
CODE_SERVER_SHA256_AMD64  published release digest
CODE_SERVER_SHA256_ARM64  published release digest
```

The version and hashes are defaults in source, not secrets. `TARGETARCH` maps only `amd64` and
`arm64`; another value exits non-zero with a clear message.

### Environment

```text
PLAYWRIGHT_SKIP_BROWSER_DOWNLOAD=1
DEBIAN_FRONTEND=noninteractive
PATH includes the copied Node `/usr/local/bin`
```

No application key, Cloudflare credential, editor password, git credential, or agent token is
defined.

### Installation order

1. Copy `/usr/local` from `node-runtime`.
2. Assert `node --version` equals `v24.18.0` during the build.
3. Select, download, and SHA-256 verify the code-server archive.
4. Extract code-server into `/opt/code-server` and link its executable.
5. Create `/opt/demo-runway`.
6. Copy `package.json` and `package-lock.json` only.
7. Run `npm ci` to produce the cached dependency layer.
8. Copy the real runtime/development project files.
9. Assert Node, code-server, and critical locked dependencies are available.
10. Set `WORKDIR /workspace`.
11. `EXPOSE 4321 8080`.

There is no new `ENTRYPOINT` or `CMD`; the final image inherits the Sandbox base entrypoint.

### Project files copied

The runnable baseline under `/opt/demo-runway` contains:


Tests and Playwright config are not required for server readiness and do not enter the final
image. Operational scripts are included because a collaborative worktree is expected to retain
the repository's real commands, but no script is run automatically at container entry.

## `src/session-worker.ts`

Public module exports:

```ts
export { Sandbox } from '@cloudflare/sandbox';
export default handler;
```

The default handler conforms to `ExportedHandler<SessionWorkerEnv>` and returns JSON containing:

```json
{
  "service": "demo-runway-sessions",
  "status": "ok"
}
```

The response is passive. It does not resolve a Sandbox ID, start a container, expose a port,
proxy traffic, accept a revision, or mutate state.

Unknown paths may return the same health document in this minimal entrypoint because routing is
not yet a public contract. `T-004-03-02` will replace this with lifecycle/hostname dispatch.

No handwritten environment interface is introduced. The generated `SessionWorkerEnv` is the
only binding contract.

## Generated type file

`worker-configuration.sessions.d.ts` is produced by:

```text
wrangler types worker-configuration.sessions.d.ts
  --config wrangler.sessions.jsonc
  --env-interface SessionWorkerEnv
  --include-runtime=false
  --strict-vars=false
```

It should contain a `Sandbox` Durable Object namespace binding. It is referenced where needed
without changing `src/env.d.ts`, whose existing reference stays tied to the stable App Worker.

The generated file is committed so reviewers and TypeScript can see the configuration contract.

## `scripts/session-image-check.ts`

### Public pure helpers

Export small helpers for unit tests:

- `parseArgs(argv)` — accepts `--image`, `--dockerfile`, `--budget-ms`, `--skip-build`;
- `parsePublishedPort(output)` — extracts the random loopback port from Docker output;
- `validateVersions(observed, expected)` — names a Node/code-server mismatch;
- `validateEntrypoint(config)` — requires `/container-server/sandbox`;
- `formatEvidence(result)` — stable JSON-compatible record.

Invalid/unknown flags and non-positive budgets fail before Docker mutation.

### Impure orchestration

The CLI uses `node:child_process` with argument arrays, never shell interpolation, for:

- `docker info`;
- `docker build --file Dockerfile.session --tag <image> .`;
- `docker image inspect`;
- diagnostic `docker run --rm --entrypoint /bin/sh`;
- detached cold container start;
- `docker port`;
- `docker logs`;
- forced `docker rm --force` cleanup.

The cold container overrides the entrypoint solely to execute:

```text
cd /opt/demo-runway
exec npm run dev -- --host 0.0.0.0 --port 4321
```

Port 4321 is published to a random loopback port. The script polls with bounded `fetch` calls,
requires HTTP 200, and verifies the page contains the project's stable title/marker.

The clock begins immediately before detached `docker run` and stops at the first valid real-app
response. The hard deadline defaults to `60_000` ms.

All failure paths include a short diagnostic and logs where available. A `finally` block removes
the named container. The generated name includes process ID and timestamp to avoid collisions.

The CLI prints one final JSON object to stdout so it can be transcribed into evidence without
parsing human prose. Progress messages go to stderr.
