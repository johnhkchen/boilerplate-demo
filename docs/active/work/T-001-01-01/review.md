## Open concerns / flags for human attention

1. **Astro security advisories (medium priority).** `npm audit` reports 2
   issues (1 high, 1 low) against Astro ≤ 7.0.0-beta.6 and its bundled esbuild.
   The high-severity items are XSS via `define:vars`, server-island parameter
   replay, unescaped slot names, spread-prop attribute names, and host-header
   SSRF on prerendered error pages. **None of these code paths are used** by
   this static single page (no `define:vars`, no server islands, no slots, no
   spread props, no custom error page). The esbuild item is dev-server-on-Windows
   arbitrary file read only. The advertised fix is `astro@7` — a **breaking
   major** outside this ticket's scope. *Recommendation:* handle the 5→7 upgrade
   as its own tooling ticket, verifying the static-build contract and the -02
   Cloudflare adapter still hold; do not force-upgrade inline here.

2. **Fonts are declared but not loaded.** The page names Lora/Karla in its
   `font-family` stacks with system-serif/sans fallbacks but ships no web-font
   files or `@link` (deliberate — no network dependency at scaffold stage per
   design D6). Real Lora/Karla rendering, matching b28.dev, is a follow-up when
   the design system earns vendoring (`b28-clay.css` / `just sync-kit`).

3. **`^5.13.0` caret range.** `package.json` allows 5.x minor/patch bumps;
   `package-lock.json` pins the exact resolved version for reproducibility. Fine
   for a demo scaffold. Tighten only if a future minor breaks the static
   contract.
