# Research — T-004-04-01 access-protects-private-surfaces

## Ticket state and required outcome

- The ticket starts in `phase: research` and requires all six RDSPI phases.
- Its acceptance criterion joins four observable properties:
  - unauthenticated requests cannot reach the session preview;
  - unauthenticated requests cannot reach the session editor;
  - `demo.b28.dev` remains anonymously reachable;
  - the origin validates the Access assertion independently of edge enforcement;
  - an invited identity is authorized only for the intended private surface;
  - revocation removes that authorization.
- The ticket advances charter principles P3 and P4: safe room/phone operation and
  browser-only collaboration.
- The ticket depends on `T-004-03-02`, whose implementation and review are present.
- The ticket explicitly forbids changing its own `phase` or `status` frontmatter.

## Repository-level trust boundaries

- `wrangler.jsonc` defines the stable public App Worker named `demo-runway`.
- That Worker owns only the exact custom domain `demo.b28.dev`.
- It serves the promoted static application and on-demand application APIs.
- It deliberately keeps `workers_dev: true` and `preview_urls: true` for stable-app health
  and pre-promotion checks.
- `wrangler.sessions.jsonc` defines a different Worker named `demo-runway-sessions`.
- The Sessions Worker owns two exact custom domains:
  - `demo-session.b28.dev` for Astro/Vite preview;
  - `code-session.b28.dev` for code-server.
- The Sessions Worker and App Worker have separate Wrangler configurations, bindings,
  deployment histories, and request handlers.
- The public App Worker has no import or binding to the Sessions Worker.
- The Sessions Worker has no import or binding to the public App Worker.
- The decision record requires this physical separation so a private-surface auth bypass
  cannot become a public-demo auth bypass.

## Current Sessions Worker request flow

- `src/session-worker.ts` is the Sessions Worker entrypoint.
- Every request first parses session routing configuration.
- `/__session/up`, `/__session/status`, `/__session/logs`, and `/__session/down` are
  classified as control operations.
- Control operations are dispatched to the singleton `SessionCoordinator` Durable Object.
- Requests whose hostname exactly matches the configured preview or editor host are
  classified as proxy requests.
- The Worker writes its own internal target header before forwarding to the coordinator.
- The coordinator removes that routing header before proxying to the container.
- Preview traffic is sent to Astro on port 4321.
- Editor traffic is sent to code-server on port 8080.
- HTTP and WebSocket traffic share the same host classifier and coordinator boundary.
- The coordinator checks lifecycle readiness and actual process state before proxying.
- Requests for other hostnames receive the service root response or a JSON 404.
- No current branch performs authentication or authorization.
- No current branch reads `Cf-Access-Jwt-Assertion`.
- No current branch validates a JWT signature, issuer, audience, or lifetime.

## Current exposure and bypass paths

- `wrangler.sessions.jsonc` currently sets `workers_dev: true`.
- It also sets `preview_urls: true`.
- Cloudflare Access applications attached only to the two branded hostnames do not, by
  themselves, protect the Sessions Worker's `workers.dev` hostname or version preview URLs.
- The Worker uses the request hostname to select a proxy target, so those alternate hostnames
  do not currently classify as editor or preview.
- They still expose the control API and service root because control classification is path
  based and occurs before hostname classification.
- Future routing changes could make an enabled alternate hostname a private-surface bypass.
- The lifecycle review explicitly flags `workers_dev: true` as an alternate-origin concern.
- The public App Worker's alternate URLs are independent and are intentional for its public
  deployment workflow.

## Current code-server posture

- This is documented as acceptable only behind the future Access boundary.
- There is no second code-server password or cookie barrier.
- A request that reaches editor port 8080 is therefore inside the collaboration trust zone.
- The preview runs repository code and also belongs inside that trust zone.
- Both HTTP navigations and WebSocket upgrades need the same authorization result.

## Cloudflare Access request contract

- Cloudflare documents that Access checks the `CF_Authorization` application cookie at the
  edge for protected HTTP applications.
- After successful Access authorization, Cloudflare forwards the application token to the
  origin in `Cf-Access-Jwt-Assertion`.
- Cloudflare recommends origin validation of that header rather than relying on the browser
  cookie, because the cookie is not guaranteed to be forwarded.
- The header value is a signed JWT.
- Merely checking for the header is insufficient and permits spoofing.
- The documented Access signing algorithm is RS256.
- Signing keys are account-specific.
- The current and previous public signing keys are published as JWKs at
  `<team-domain>/cdn-cgi/access/certs`.
- Access rotates signing keys by default and retains the previous key during a transition.
- Cloudflare recommends remote key retrieval keyed by the JWT `kid`, rather than embedding
  one certificate.
- Cloudflare's current Worker example uses `jose`, `createRemoteJWKSet`, and `jwtVerify`.

## Application-token claims

- An identity application token has an `iss` claim equal to the Cloudflare Access team
  domain URL.
- Its `aud` claim contains the Access application's Application Audience tag.
- Each Access application receives a unique audience tag.
- The audience remains stable unless the application is deleted and recreated.
- Its `exp` claim bounds token validity.
- Its `nbf` claim can prevent use before the issued validity window.
- Its `type` is `app`; the global organization token has a different purpose.
- Identity tokens contain an IdP-verified `email` and an account-scoped `sub`.
- Service-token application tokens need not contain an email.
- The ticket calls for identity-login-only collaboration, so the relevant successful token is
  an identity application token rather than a service-token token.

## Per-surface authorization facts

- Access policy authorization happens before the request reaches the Worker.
- Access is default-deny when an application has no matching Allow policy.
- An Allow policy needs at least one Include selector.
- An exact Email selector can represent an explicit invitation for this one-session MVP.
- Broad `Everyone` or all-valid-email selectors would not represent invitation-only access.
- A single multi-domain Access application can cover both private hostnames.
- Such an application uses one application authorization boundary and one audience.
- Separate Access applications provide separate audience tags and separately managed Allow
  policies.
- The acceptance criterion's “only its intended surfaces” requires the origin to distinguish
  preview authorization from editor authorization.
- Exact host classification already supplies the surface selection needed for that distinction.

## Revocation behavior

- Access application tokens remain usable for their configured session lifetime unless
  revoked or expired.
- Per-application token revocation ends active sessions for all users of that application.
- Per-user revocation ends that user's Access sessions across applications.
- Permanently removing access also requires disabling/removing the identity in the IdP or
  removing it from the relevant Allow policy, so it cannot immediately obtain a new token.
- Cloudflare documents a short propagation window for logout/revocation effects.
- Origin JWT validation naturally rejects an expired token.
- Origin audience validation naturally rejects a still-valid token minted for the other
  private application.
- The repository has no Cloudflare account credentials or IdP integration details with which
  to execute a real remote revocation test.

## Policy and provisioning ownership

- Wrangler custom-domain routes bind hostnames to Workers but do not define Zero Trust Access
  applications or their identity policies.
- The repository currently has no Terraform configuration or Access-provisioning script.
- It has no account ID, Access application IDs, policy IDs, team domain, or invited email list.
- The decision record says automation should manage DNS/Access after a one-time zone setup.
- The current MVP has exactly one fixed session and two fixed private hostnames.
- A deployable repository contract therefore needs account-specific Access values supplied at
  deployment time rather than committed from another account.

## Research constraints summary

- The public and private Workers are already physically separate.
- The private Worker has exact surface routing but no authentication.
- code-server itself has no authentication.
- Access must protect both branded private hostnames at the edge.
- Origin code must verify signature, issuer, audience, validity, and identity token shape.
- Preview and editor require different audience expectations to enforce surface isolation.
- Alternate private Worker hostnames must not create a route around hostname-scoped Access.
- Account-specific identities and audience tags cannot be inferred from the repository.
- Local cryptographic and routing tests can be complete; remote policy/revocation proof depends
  on an operator with Cloudflare and IdP authority.
