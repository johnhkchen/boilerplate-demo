_Proposed by Vend's `propose-epic` play._

## Intent — the bigger-picture play

The whole check harness is wired to the receipt exemplar, not to the seam it is supposed to guard: integration-check hardcodes `/api/receipt` (OPS_CHECK_URL/LEAK_CHECK_URL), ops-check and leak-check default to `${baseUrl}/api/receipt`, the signature verification assumes the `DEMO_SIGNING_KEY`/receipt signing scheme, and the healthy Playwright flow asserts the literal `Demo Runway` heading plus receipt-shaped DOM (`#receipt-nonce` 32-hex, `#receipt-signature` 64-hex, `#receipt-body`). So Day-1's core assembly move — replacing the placeholder slice behind the boundary with a real demo — quietly guts every check: they keep going green against a route and page that no longer exist, or go red for reasons that have nothing to do with the real boundary's health. The bigger-picture play is to invert the coupling: let a boundary DECLARE what to check (its path, the response contract/shape it must satisfy, its signing/verification scheme, and the page landmark the flow should wait on), and have the harness read that declaration generically instead of naming the receipt exemplar. The receipt then becomes just the first boundary that fills in the contract, not the thing the checks are built around — so a new demo swaps the slice, updates one declaration, and inherits meaningful checks for free.

## Value to the design

This makes P2 hold for the REAL demo, not just the shipped exemplar: the checks that prove the core moment end to end, surface stalled boundaries, and keep secrets out of bundles stay wired to whatever the current boundary actually is, so a slice swap can't leave the harness asserting a moment that is no longer on the page. It advances P1 by protecting the Day-1 assembly path — the move that turns a fresh repo into a working public demo no longer silently invalidates its own safety net, so integration failures still surface fast during exactly the step where they're most likely. As a permanent it converts a hand-maintained, exemplar-shaped harness into a reusable declared contract every generated demo inherits (an admission-test 'reliable seam that still allows idea-driven implementation'), so the leverage compounds across demos instead of being re-derived per project.

## Done looks like

Replacing the slice behind the seam with a different boundary — a new path, a differently-shaped signed response, and a different page heading — and updating only that boundary's declaration (no edits to the check scripts or the flow spec) yields checks that still pass when the new boundary is healthy AND fail explicitly when it is broken, stalled, or leaks; conversely, deleting the old `/api/receipt` route without updating the declaration makes the harness fail loudly (a missing/￢unsatisfied contract) rather than passing green against a route that no longer exists. The receipt is expressed as one instance of the generic contract, and `npm run verify` still passes end to end on the receipt boundary as the worked example.

## Context & constraints

Go-and-see: the coupling lives in scripts/integration-check.ts (commandFor pins `/api/receipt` into OPS_CHECK_URL/LEAK_CHECK_URL and threads DEMO_SIGNING_KEY), scripts/ops-check.ts and scripts/leak-check.ts (default URL `${baseUrl}/api/receipt`, DEMO_SIGNING_KEY verification), the pure cores under src/lib/{integration-check,ops-check,leak-check}.ts, and tests/demo-flow.spec.ts (literal `Demo Runway` heading, receipt-shaped DOM, PRIMARY_ACTION_NAME, `**/api/receipt` route). The declaration this epic introduces is a portable in-repo contract the boundary owns — NOT an all-provider abstraction layer (N2: one boundary contract earned by the receipt's repeated use, not a speculative provider matrix) and NOT a SaaS control plane (N3). It is a machine-checkable contract replacing exemplar-shaped assertions (admission test: 'replaces stale explanatory state with machine-checkable contracts'), and the checks still only catch hangs/regressions/leaks — they do not judge whether the demo is convincing (N4, N5). Prerequisites, named as such: the boundary needs a place to declare its path/shape/signing/landmark that both the Node scripts and the Playwright config can read — the existing DEMO_BASE_URL / *_CHECK_URL / PLAYWRIGHT_BASE_URL env seam and playwright.config.ts projects are the seam to extend, not replace. Out of scope: changing what the receipt boundary DOES, adding new boundaries, the session/backstage checks beyond making the shared harness generic, and any provider abstraction. Right-size (PE-7): defining the boundary-contract shape + rewiring the three scripts and the healthy flow to read it + proving it with a swap test fits one ~2-hour block; a fleet-wide 'every demo ships a contract linter' gate would be a separate epic, not hidden scope here.
