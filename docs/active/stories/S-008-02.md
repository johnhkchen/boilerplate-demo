**Scope:** The gated HTTP surface over the new data layer. Evolve the agent-retrieval read seam — src/lib/backstage-retrieval.ts + its feed envelope + src/pages/api/backstage/feed.ts — so every entry now carries its stable id and completion state verbatim, oldest-first, and reflects completes and deletes. Add the management operations as a new framework-free core (src/lib/backstage-management.ts, mirroring backstage-route.ts) behind a new dynamic edge src/pages/api/backstage/entries/[id].ts (PATCH completion, DELETE), each composing the existing guardPasscode. The submit route (POST /api/backstage/entries) is untouched.

**Story acceptance:** A Node test injecting a SQLite store shows: GET feed returns entries whose id and completion match the store byte-for-byte and reflect a just-completed and a just-deleted entry; npm run backstage:feed still prints them. A route test shows PATCH complete and DELETE by id succeed only with the valid passcode and change exactly the addressed entry, while a wrong passcode or unknown id is refused with a distinct status and the store is unchanged — the gate is the outer wall on every mutation, never bypassed.

**Honest boundary:** Gate and store are composed and unit-tested in Node against an injected in-process store — FREE, no live Worker, no Cloudflare env. The deployed routes behind cloudflare:workers env are exercised by the e2e story (S-008-03), named here, not claimed by this story. The seam gains id+completion only; it adds no editing, threading, moderation, or agent write-back — the agent stays read-only on the seam.
