**Scope:** The single passcode-gated view that folds the submit-only form and the read seam into one surface. Rewrite src/pages/backstage.astro (+ its inline client script) into an unlock-once dashboard: entering the shared passcode fetches the gated feed client-side; on success the page reveals a checklist that submits new entries (POST /api/backstage/entries), marks entries complete (PATCH .../[id]), and deletes them (DELETE .../[id]) — all reusing the in-memory passcode header. Plus the end-to-end proof in tests/backstage-flow.spec.ts (Playwright backstage project).

**Story acceptance:** Given only the URL and passcode, a visitor unlocks once and in one view submits an entry, sees every existing entry as a checklist alongside it, marks one complete, and deletes another — with no account and no second credential; a wrong passcode refuses the unlock. leak:check confirms the passcode never enters the bundle. The Playwright backstage project drives unlock → submit → complete → delete and asserts the agent feed reflects each change; npm run test:flow:backstage is green.

**Honest boundary:** The page and its client JS are proven by the Playwright backstage project against a local server — the deployed-dashboard smoke belongs to the deploy step, not this story. Unlock holds the passcode in page memory only: no cookie, no session record, no account, no per-user identity is introduced (N5 — not framework-by-inertia) — the page stays a static edge asset whose only server contact is the gated /api/* seams (P3 — the room and phone both work safely).
