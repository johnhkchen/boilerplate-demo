**Scope:** The pure data layer that every management operation stands on: a new append-only D1 migration (migrations/0002_*.sql) adding a completion column to backstage_entries, the portable contract in src/lib/backstage-entry.ts (BackstageEntry gains a stable public handle + completion state), and src/lib/backstage-store.ts (list projection now carries id+completion; new complete and delete operations by handle). The E-003 id column already exists as INTEGER PRIMARY KEY but is deliberately unexposed — this story makes it public because a checklist entry cannot be completed or deleted without an addressable handle.

**Story acceptance:** Against the committed migration in in-process SQLite (backstage-store.test.mjs idiom), a saved entry round-trips carrying a stable id and an uncompleted default; setting completion by that id flips exactly that row and clears back; deleting by that id removes exactly that row and no other; listEntries stays oldest-first (id ASC) and still drops no physical column beyond the settled public set. npm test and npm run typecheck are green against the new shape.

**Honest boundary:** Everything here is pure and FREE — Node's SQLite against the real committed migration, no Cloudflare, no Worker, no tokens. Applying migration 0002 to the deployed D1 database is a metered operator step at deploy time, named here, not hidden in a ticket. Deliberately no soft-delete/undo and no audit history (epic out-of-scope): delete is a hard row removal, completion is a settable flag with no prior-state trail.
